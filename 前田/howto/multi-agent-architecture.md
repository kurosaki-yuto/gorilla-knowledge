# マルチエージェントシステムの設計パターン

## 概要

複数のAIエージェントを協調させてタスクを処理するシステムの設計パターン。実運用で効果的だった構成をまとめる。

## 3層ヒエラルキーモデル

```
戦略層（CEO / COO）
  └─ 意思決定・優先順位判断
  └─ モデル: 高性能（Opus等）を控えめに使う

調整層（CoS / Coordinator）
  └─ タスク分解・アサイン・進捗管理
  └─ モデル: 中性能（Sonnet等）

実行層（Developer / Writer / Researcher等）
  └─ 実装・コード生成・文書作成
  └─ モデル: 高速（Haiku等）
```

### ポイント

- **戦略層は実装しない**。意思決定だけ。コード書かせると高コスト＆過剰品質
- **調整層は実行しない**。分解して渡すだけ。自分でやり始めると詰まる
- **実行層は判断しない**。言われたことを正確にやる。判断は上に返す

## Confidence-Based Execution（信頼度ベース実行）

タスクを受けたとき、信頼度で実行判断を分岐する：

| 信頼度 | アクション |
|--------|-----------|
| High（確信あり） | そのまま実行。完了報告のみ |
| Medium（概ね分かる） | 仮説を提示して承認を待つ |
| Low（不明確） | 質問して要件を明確化してから再判断 |

**よくある失敗**: 全タスクを質問から始めると「質問ばかりで何も進まない」エージェントになる。Highのものは即実行すべき。

## エージェント間通信のルール

- **Discordスレッド** を軽量な通信チャネルとして使う（チャンネル = チーム、スレッド = タスク）
- 正本データはDiscordに置かない。DBに書いてURLだけ共有する
- エージェント追加のルール：
  - 調整層（CoS）配下 → 自由に追加OK、事後報告
  - 戦略層（CEO）直下 → 人間の承認必須

## 1エージェント = 1責務

- 役割の重複を禁止する
- 「何でもできるエージェント」は作らない
- 新しい責務が必要になったら、既存エージェントに足すのではなく新しく作る

## モデル選定の指針

| 用途 | 推奨モデル | 理由 |
|------|-----------|------|
| 戦略的判断 | Opus | 精度重視。使用頻度は低くてOK |
| タスク分解・調整 | Sonnet | バランス。1日に何度も呼ぶ |
| コード生成・文書作成 | Haiku | 速度重視。大量に呼ぶ |
| 単純な変換・フォーマット | Haiku | コスト最小化 |
