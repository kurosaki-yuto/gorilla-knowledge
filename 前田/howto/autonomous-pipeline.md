# 自律実行パイプラインの作り方

## 概要

複数のAIエージェントを順番に起動して、人間の介入を最小限にしながらタスクを完了させるパイプラインの設計パターン。

## 基本構造

```
要件定義エージェント
  ↓ requirements.md
設計エージェント
  ↓ design.md
実装エージェント（フロント/バック並列可）
  ↓ コード
テストエージェント
  ↓ テスト結果
デプロイエージェント
  ↓ デプロイ完了通知
```

## 設計のポイント

### 1. フェーズゲート方式

各フェーズは前のフェーズが完了してからしか開始しない。MVPフェーズでは**並列実行しない**（依存関係が不明確なうちは逐次実行の方が安全）。

### 2. ファイルベースの受け渡し

エージェント間のデータ受け渡しは**ファイル経由**で行う。APIやメモリ共有ではなく、`requirements.md` → `design.md` → `plan.md` のようにファイルに書き出す。

**理由**:
- 人間がいつでも中間成果物を確認できる
- エージェントが途中で落ちても、ファイルが残っているので再開可能
- デバッグが容易（どのフェーズで何が出力されたか追跡できる）

### 3. 進捗ファイルで状態管理

```json
// pipeline-progress.json
{
  "current_phase": "implementation",
  "phases": {
    "requirements": { "status": "completed", "completed_at": "..." },
    "design": { "status": "completed", "completed_at": "..." },
    "implementation": { "status": "in_progress", "started_at": "..." },
    "testing": { "status": "pending" },
    "deployment": { "status": "pending" }
  }
}
```

### 4. ハートビート監視

15分ごとに以下をチェック：
- パイプラインが停止していないか
- 承認待ちのタスクがないか
- エラーが蓄積していないか

### 5. 承認ポイントの設計

全てを自動化しない。以下のタイミングで人間の承認を挟む：
- 要件定義の完了後（方向性の確認）
- デプロイ前（本番反映の最終確認）

**失敗例**: 承認ポイントが多すぎると、人間がボトルネックになってパイプラインが止まる。2-3箇所に絞る。

## スケジュール例（6時間MVP）

```
10:00 - 要件定義（30分）
10:30 - 人間が承認（15分想定）
10:45 - 設計（45分）
11:30 - 実装（3時間）
14:30 - テスト（45分）
15:15 - 人間が承認（15分想定）
15:30 - デプロイ（30分）
16:00 - 完了
```

## 注意点

- **最初は逐次実行から始める**。並列化は安定してから
- **中間ファイルは上書き、バージョン管理しない**（git管理は最終成果物だけ）
- **エージェントのスポーン時にタイムアウトを設定する**。無限ループ防止
