# 3段階エラーエスカレーション設計

## 概要

AIエージェントが自律的に動くシステムでは、全エラーを人間に通知すると通知疲れが起きる。かといって全て無視すると重大な問題を見逃す。3段階に分けることで、適切なバランスが取れる。

## 3段階の設計

### Level 1: 自動修復（通知なし）

**対象**: レートリミット、ネットワーク一時障害、構文エラー

**対処**: Exponential Backoff でリトライ

```
1回目: 5秒後にリトライ
2回目: 10秒後
3回目: 20秒後
4回目: 40秒後
5回目: 80秒後
→ それでも失敗 → Level 2 へ
```

**ポイント**: このレベルのエラーは日常的に起きる。いちいち通知しない。

### Level 2: 3回試行 → エスカレーション

**対象**: デプロイエラー、テスト失敗、認証エラー

**対処**:
1. エラーログを分析
2. 自動で修正案を生成
3. 修正を適用してリトライ
4. 3回失敗したら Level 3 へ

### Level 3: 即座に人間へ通知

**対象**: 解決不能なエラー、仕様変更が必要な問題、3回以上修正に失敗

**対処**: Discordの `#alert` チャンネルに通知
- エラー内容
- 試行した対処法
- 必要な判断事項

## 監視メトリクス

| 指標 | 閾値 | アクション |
|------|------|-----------|
| 開発中タスクの無更新時間 | 2時間以上 | アラート |
| 承認待ち残り時間 | 2時間以下 | リマインダー |
| 同一エラーの発生回数 | 3回以上 | Level 3 へ |
| 長期承認待ち | 48時間以上 | 自動リマインダー |

## 実運用での教訓

- **Level 1 を飛ばして全部通知** → 1日100件以上の通知が来て誰も見なくなる
- **Level 3 の通知にコンテキストがない** → 「エラーが出ました」だけでは判断できない。何を試して何が失敗したかを必ず含める
- **アラート頻度のレートリミット** → 同じエラーを毎分通知しない（失敗事例あり）
