# カスタムスキルの作り方

## スキルのファイル構造

```
skills/
└── my-skill/
    ├── SKILL.md           ← 必須（スキル定義）
    ├── scripts/           ← オプション（実行スクリプト）
    ├── references/        ← オプション（参考資料）
    └── assets/            ← オプション（テンプレート等）
```

## SKILL.mdの書き方

### フロントマター（メタデータ）

```yaml
---
name: my-skill
description: このスキルが何をするか（1行で簡潔に）
read_when: どういう指示で起動するかのキーワード
metadata:
  version: 1.0
  author: your-name
---
```

**read_when が重要**: Claude Codeは全スキルのread_whenを見て、ユーザーの指示に合うスキルを自動選択する。曖昧だと誤起動する。

### 本文（手順）

```markdown
## 概要
（1-2文でスキルの目的を説明）

## 手順
1. ○○を確認する
2. △△を実行する
3. 結果を□□に保存する

## 入力
- テーマ（必須）
- フォーマット（オプション、デフォルト: Markdown）

## 出力
- workspace/output/YYYY-MM-DD/{ファイル名}
```

## 設計の3原則

### 1. 短く書く

```
❌ 500行のSKILL.md → コンテキストを圧迫
✅ 100行以内 → 必要な指示だけ
```

コンテキストウィンドウは全員で共有する資源。スキル定義が長いと、他の情報（コード、会話履歴）が押し出される。

### 2. 構造化する

```
❌ 「まず○○を確認してください。確認が終わったら、
    次に△△を実行してください。その際、□□に注意して...」

✅ Step 1: ○○を確認
   Step 2: △△を実行（注意: □□）
   Step 3: 結果を保存
```

箇条書き・テーブル・番号付きリストを使う。

### 3. 自由度をコントロールする

| タスクの性質 | 指示の詳細度 |
|-------------|-------------|
| 創造的（文章作成等） | 方針だけ指定、詳細はAIに任せる |
| 決定的（デプロイ等） | 具体的なコマンドをスクリプトで指定 |
| 半構造的（調査等） | 手順は指定、内容の判断はAIに任せる |

## スクリプトの同梱

AIが苦手な処理（ファイル変換、API呼び出し等）はスクリプトで補う：

```
skills/my-skill/
├── SKILL.md
└── scripts/
    ├── convert.py       ← Pythonスクリプト
    └── deploy.sh        ← シェルスクリプト
```

SKILL.md内で参照：
```markdown
## Step 3: 変換
scripts/convert.py を実行して出力をPDFに変換する
```

## 参考資料の遅延読み込み

大量のドキュメントはreferences/に分離して、必要なときだけ読み込む：

```
skills/my-skill/
├── SKILL.md                    ← 常に読まれる（短くする）
└── references/
    ├── api-reference.md        ← API関連の質問時のみ読む
    ├── design-guide.md         ← デザイン関連時のみ読む
    └── troubleshooting.md      ← エラー時のみ読む
```

SKILL.md内で指示：
```markdown
## APIの詳細が必要な場合
references/api-reference.md を参照すること
```

## よくある失敗

### 作りすぎ問題
41スキル作って実際に使ったのは6つ。「いつか使うかも」で作ったスキルは使われない。

**対策**: 3回以上同じ作業をしたら初めてスキル化を検討する。

### トリガーの衝突
複数のスキルのread_whenが似ていると、意図しないスキルが起動する。

**対策**: read_whenは具体的に書く。「レポート」ではなく「週次営業レポートを生成」のように。

### ドキュメントだけスキル
SKILL.mdを書いたが、実際に動かしたことがない。動かすと想定通りに動かないことが多い。

**対策**: スキルを作ったら必ず1回テスト実行する。
