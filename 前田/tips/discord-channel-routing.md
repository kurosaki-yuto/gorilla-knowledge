# Discordチャンネルルーティングの設計パターン

## 基本構造

Discordのチャンネルごとに異なるAIエージェントを割り当てる設計。

```
#general     → メインエージェント（何でも対応）
#dev         → 開発エージェント（コード・技術質問）
#content     → ライターエージェント（文章作成）
#alert       → 監視エージェント（エラー通知受信）
#feed        → 通知専用（エージェントからの報告）
#log         → ログ専用（デバッグ・監査用）
```

## チャンネルプロファイルの設計

各チャンネルに「プロファイル」を持たせて、動的にスキルをロードする：

```json
{
  "channel_id": "1234567890",
  "agent": "developer",
  "skills": ["code-review", "debug", "deploy"],
  "priority": "high",
  "auto_save": true
}
```

- **skills**: そのチャンネルで使えるスキルを限定する
- **priority**: 同時リクエスト時の処理優先度
- **auto_save**: 成果物を自動保存するかどうか

## プロファイルのキャッシュ

Notion（またはDB）からプロファイルを読み込み、1時間キャッシュする。毎回APIを叩かない。

## よくある落とし穴

### 1. 暗黙のルーティング失敗
複数チャンネルが設定されている場合、「どのチャンネルに返信するか」を明示的に指定しないとエラーになる。

### 2. チャンネルが見つからないエラー
チャンネルIDのハードコードは避ける。設定ファイルから読み込む。チャンネル削除・再作成でIDが変わる。

### 3. 用途別チャンネル分離の重要性
通知と会話を同じチャンネルに混ぜると、重要な通知が流れてしまう。最低限 `#alert`（重要通知）と `#log`（ログ）は分ける。
