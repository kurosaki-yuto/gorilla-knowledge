# AIエージェントのメモリ管理ベストプラクティス

## 問題

AIエージェント（LLMベース）はコンテキストウィンドウに上限がある。メモリ管理を怠ると：
- 起動が遅くなる
- 重要な情報が押し出されて忘れる
- トークンコストが増大する

## 設計原則

### 1. 起動時に読み込むファイルは最小限にする

**目標: 5,000トークン以下**

```
❌ SOUL.md (35KB) + AGENTS.md (8KB) + ログ全部 = 17,400トークン
✅ SOUL.md (8KB) + 要約のみ = 5,000トークン以下
```

### 2. 3層メモリ構造

```
hot/     → 常にコンテキストに入る（プロンプト、基本設定）
warm/    → 必要時に読み込む（今日のタスク、直近のログ）
cold/    → 検索時のみアクセス（過去のログ、アーカイブ）
```

### 3. 自動アーカイブ

- 7日以上前のログは `archive/` に移動
- 30日以上前のアーカイブは要約だけ残して削除
- 文字起こしログは `summary.md`（要約）+ `raw/`（全文、検索用）に分離

### 4. メモリの一元管理

メモリを複数箇所に分散させない。

```
❌ root/memory/ + workspaces/memory/ + executive/ceo/memory/（3箇所に分散）
✅ memory/（1箇所に集約）
```

分散すると「どこに何があるか分からない」問題が起きる。

## SQLiteメモリのサイズ管理

SQLite（ベクトル検索用）が肥大化すると検索が遅くなる。

- 68MBを超えたら要注意
- 古いエントリの定期パージを入れる
- SQLiteのブルートフォースcos類似度検索は~100kアイテムまでが実用的。それ以上はpgvector等の専用DBを使う

## 実運用で学んだこと

- **ログは書くだけでは意味がない**。定期的に要約して古いのは消す
- **「全部記憶する」は不可能**。忘れる仕組み（アーカイブ・削除）が必要
- **メモリのサイズは定期的に計測する**。知らないうちに膨らんでいる
