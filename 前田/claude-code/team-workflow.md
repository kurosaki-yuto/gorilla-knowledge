# Claude Codeのチームワークフロー

## チームでClaude Codeを使うときの設計パターン

### 役割分離パターン（Architect → Planner → Builder）

大きなタスクを3つの役割に分けて処理する：

```
Architect（設計者）
  └─ システム全体のアーキテクチャを決める
  └─ 使用モデル: 高性能（判断精度重視）

Planner（計画者）
  └─ 設計を具体的なタスクに分解する
  └─ ファイル単位・関数単位の実装計画

Builder（実装者）
  └─ 計画に従ってコードを書く
  └─ 使用モデル: 高速（実装速度重視）
```

**1人で全部やらせると何が起きるか**:
- 設計と実装が混在して方向性がブレる
- 途中で方針が変わって手戻りが発生する
- コンテキストが溢れて前の指示を忘れる

### 実践的なフロー

```
1. planモード（/plan）で設計 → ユーザーが承認
2. 承認された計画に沿って実装
3. 実装後にレビュー
4. テスト実行 → 問題があれば修正
```

## CLAUDE.mdを共有リポジトリに入れる

チーム全員が同じCLAUDE.mdを使うことで：
- コーディング規約が自動適用される
- ファイル配置ルールが統一される
- 禁止事項が共有される

```
repo/
├── CLAUDE.md          ← チーム共通のルール
├── src/
├── tests/
└── ...
```

## メモリの活用（.claude/memory/）

Claude Codeはセッション間で記憶を保持できる。チームで有効な使い方：

- **ユーザーメモリ**: 自分の得意分野・好みのコーディングスタイルを記憶させる
- **フィードバックメモリ**: 「テストはmockを使わない」等の指摘を記憶させると、次回から従う
- **プロジェクトメモリ**: 進行中のプロジェクトの状態を記憶

```
「このプロジェクトではテストにvitestを使っている。覚えておいて」
→ 次回以降、テストを書くときにvitestで書いてくれる
```

## Agentモード（サブエージェント）の活用

複数の独立した調査を同時に走らせる：

```
「以下を並列で調査して：
  1. src/api/ のエンドポイント一覧
  2. tests/ のカバレッジ状況
  3. package.json の依存関係の脆弱性」
```

Claude Codeがサブエージェントを起動して並列処理する。待ち時間が大幅に短縮される。

## コードレビューの活用

```
「このPRの変更をレビューして。特に以下を確認：
  - セキュリティリスク（SQLインジェクション、XSS等）
  - パフォーマンス問題
  - テストの網羅性」
```

人間のレビュー前にClaude Codeでプレレビューすると、基本的な問題を事前に潰せる。

## 注意点

- **全員が同じCLAUDE.mdを使う** ことで出力の一貫性を保つ
- **コンテキストの上限を意識する**: 1セッションに詰め込みすぎない
- **途中経過を確認する**: 特に大きな変更は段階的にチェック
- **機密情報を渡さない**: APIキー、パスワード等はClaude Codeに渡さず、環境変数で管理
