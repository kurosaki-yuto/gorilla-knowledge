# CLAUDE.mdの設計パターン

## CLAUDE.mdとは

プロジェクトのルートに置くファイルで、Claude Codeの振る舞いを定義する。人間のチームにおける「チームルール」や「コーディング規約」に相当する。

## なぜ必要か

CLAUDE.mdがないと：
- 毎回同じ指示を繰り返す必要がある
- エージェントが勝手な判断をする（ファイル構成を崩す、不要なファイルを作る等）
- メンバーごとにClaude Codeの動きが違う

## 効果的なCLAUDE.mdの構成

```markdown
# プロジェクト名

## ディレクトリ構成
（どこに何を置くかを明示する）

## 絶対禁止事項
（やってはいけないことをリスト化）

## 必須ルール
（守るべきルールを具体的に書く）

## よくある判断基準
（迷ったときの判断テーブル）
```

### 各セクションのポイント

### 1. ディレクトリ構成

```markdown
project/
├── src/           ← ソースコード
├── tests/         ← テストコード
├── docs/          ← ドキュメント
├── scripts/       ← 運用スクリプト
└── config/        ← 設定ファイル
```

**効果**: Claude Codeがファイルを作るとき、正しい場所に置いてくれる

### 2. 絶対禁止事項

```markdown
❌ .envをgitにコミットしない
❌ 既存のAPIキーを変更しない
❌ テストを書かずにプッシュしない
❌ mainブランチに直接pushしない
```

**コツ**: 「やるな」だけでなく「代わりにこうしろ」も書く

### 3. 必須ルール

```markdown
- ファイル名はkebab-caseで統一する
- 新規ファイルを作る前に既存ファイルで代替できないか確認する
- エラーハンドリングを必ず入れる
```

### 4. よくある判断基準

```markdown
| 状況 | 判断 |
|------|------|
| 新機能を追加したい | まずドキュメントを確認する |
| テストが通らない | ログを確認してから修正する |
| DBスキーマを変えたい | migration fileを作成してから適用する |
```

**効果**: Claude Codeが「どうしよう？」と聞いてくる回数が減る

## 実運用で学んだこと

- **短く書く**: 長すぎるとコンテキストを食う。A4で2ページ以内が目安
- **具体例を入れる**: 抽象的なルールは守られない
- **更新する**: 運用しながら「これも禁止にすべき」を追加していく
- **チームで共有する**: 1人だけのCLAUDE.mdではなく、リポジトリにコミットして全員が同じルールで動くようにする
