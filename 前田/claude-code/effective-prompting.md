# Claude Codeで効果的な指示の出し方

## 基本原則

Claude Codeは「コードを書くAI」ではなく「チームメイト」として使うと効果的。指示の出し方で出力の質が大きく変わる。

## 信頼度ベースの指示

### やりたいことが明確なとき

```
❌ 「ログイン機能を作って」（曖昧）
✅ 「src/auth/ にJWT認証のログインAPIを作って。
    POST /api/login でemail+passwordを受け取り、
    Supabaseのusersテーブルで認証して、
    JWTトークンを返す。テストも書いて。」
```

**具体的であるほど、期待通りの出力になる。**

### やり方が分からないとき

```
❌ 「なんかいい感じにして」
✅ 「このファイルのパフォーマンスが悪い。
    まずボトルネックを特定して、
    改善案を3つ提案して。
    実装はその後で相談したい。」
```

**段階を分ける。** 調査 → 提案 → 実装の順で進めると手戻りが減る。

## 5要素を含める

効果的な指示には以下の5つが含まれている：

| 要素 | 例 |
|------|---|
| **誰のためか** | 「フロントエンドチームが使う」 |
| **何のためか** | 「APIドキュメントの自動生成」 |
| **いつまでか** | 「今日中に動くものがほしい」 |
| **品質基準** | 「そのままプルリクに出せるレベル」 |
| **制約** | 「既存のExpress構成を壊さない」 |

全部入れる必要はないが、多いほど精度が上がる。

## コンテキストの渡し方

### ファイルを読ませる

```
「src/api/users.ts を読んで、同じパターンで
 src/api/orders.ts を作って」
```

**既存のコードパターンを読ませてから新しいコードを書かせる** のが最も一貫性のある出力になる。

### 前提条件を伝える

```
「このプロジェクトはNext.js 14のApp Routerを使っている。
 DBはSupabase。認証はJWT。
 この前提でユーザープロフィール編集ページを作って。」
```

## やってはいけない指示パターン

### 1. 一度に全部やらせる

```
❌ 「ECサイトを作って」
✅ 「まず商品一覧APIを作って」→ 完了 →「次にカート機能を作って」
```

### 2. 曖昧な修正指示

```
❌ 「なんか動かないから直して」
✅ 「npm run dev でエラーが出る。エラーメッセージは〇〇。
    src/components/Header.tsx の23行目が怪しい。」
```

### 3. レビューなしで進める

```
❌ 「全部自動でやって、終わったら教えて」
✅ 「まず設計案を見せて」→ OK →「実装して」→ 確認 →「テスト書いて」
```

**チェックポイントを挟む。** 特に大きな変更は段階的に確認する。

## 実践Tips

- **「〇〇を読んでから」** をつけると、既存コードとの一貫性が保たれる
- **「テストも書いて」** を毎回つけると、テスト文化が定着する
- **「変更点を説明して」** を最後につけると、何が変わったか理解できる
- **planモードを活用する**: 大きなタスクはまず計画を立てさせてから実装
