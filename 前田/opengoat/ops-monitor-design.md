# Ops Monitor（監視エージェント）の設計

## 設計思想

「何も言ってこない = 全て正常」

監視エージェントの最大の価値は **異常時にだけ知らせてくれること**。正常時に通知してくるのはノイズ。

## ヘルスチェック項目

毎日深夜3時（JST）に自動実行：

| チェック項目 | 方法 | 異常判定 |
|-------------|------|---------|
| Gateway稼働 | ポート18790にHTTPリクエスト | レスポンスなし |
| Supabase API | REST APIにクエリ | 2秒以上のレイテンシ |
| Discord Bot | オンライン状態 + メッセージ送信テスト | オフライン or 送信失敗 |
| スタックジョブ | 24時間以上pending状態のジョブ | 存在する |
| 失敗ジョブ | 直近24時間で3件以上失敗 | 閾値超過 |
| ディスク容量 | workspace/output/ のサイズ | 10GB超過 |
| ログローテーション | 最終ローテーション日 | 7日以上未実行 |

## 3段階のアラートレベル

```
🟢 Green: 全チェック正常
   → 何もしない（沈黙）

🟡 Yellow: 警告レベル
   → 自動修復を試行（ログ圧縮、temp削除等）
   → #log に記録
   → 人間への通知なし

🔴 Red: 致命的
   → #alert に即時通知
   → CEOに報告
   → 推奨対処法を提示
```

## 自動修復の権限

Ops Monitorが自分で実行できること：

| できること | できないこと |
|-----------|-------------|
| スタックジョブをfailed状態に変更 | DBスキーマの変更 |
| ログのローテーション・圧縮 | 設定ファイルの編集 |
| tempファイルの削除 | エージェントの追加・変更 |
| ヘルスチェック結果のログ記録 | サービスの起動・停止 |

## 実運用で発生した問題

### 偽アラートの嵐

Gatewayの監視機能が「15分間活動なし」を毎分通知してきた。ハードコードされていて設定変更不可だった。

**対処**: 受信側でフィルタリング。根本解決にはGatewayの修正が必要。

**教訓**: 監視システムを導入する前に「アラート頻度の制御が可能か」を確認する。レートリミットのない監視は使えない。

### チェック項目の追加順序

最初は全チェック項目を一度に導入した。結果、false positive（偽陽性）が多発して信頼性が下がった。

**教訓**: チェック項目は1つずつ追加して、閾値を調整してから次に進む。一気に導入すると、どのチェックが信頼できるか分からなくなる。
