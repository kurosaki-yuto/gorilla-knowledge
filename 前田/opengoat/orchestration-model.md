# OpenGoat オーケストレーションモデル

## 概要

OpenGoatは3体のAIエージェントで構成されるオーケストレーション層。自分では実装せず、**意思決定・分解・監視** だけを行う。

## 3体の役割

### CEO（戦略判断）

**モデル**: Opus（高性能・高コスト）を控えめに使用

**やること**:
- ユーザーの要望を受けて **仮説を立てる**（質問ではなく提案から入る）
- 信頼度を判断して、実行に回すかどうか決める
- ユーザーとの壁打ちで方向性を固める

**やらないこと**:
- コードを書く
- ファイルを直接操作する
- ルーティンの状態報告

**コミュニケーションの原則**: 「こう思います。なぜなら〜。どう思いますか？」の形式。質問から入らない。

### CoS / Kevin（タスク分解・品質管理）

**モデル**: Sonnet（中性能・バランス）

**やること**:
- CEOから来た指示を実行可能なタスクに分解する
- 5要素チェック（後述）でタスクの完全性を確認する
- 実行結果の品質をチェックする
- 成果物の3ステップ配信（ローカル保存 → Notion記録 → Discord通知）

**やらないこと**:
- 戦略判断（CEOに返す）
- 実装作業（実行層に委譲）

### Ops Monitor / Marcus（監視）

**モデル**: Haiku（高速・低コスト）

**やること**:
- 深夜3時に自動ヘルスチェック実行
- 異常検知時のみアラート通知
- スタックしたジョブの自動マーク
- ログのローテーション

**やらないこと**:
- 正常時の報告（沈黙 = 正常）
- スキーマ変更
- 設定ファイルの編集

## Confidence-Based Execution（信頼度ベース実行）

CEOがリクエストを受けたときの判断フロー：

```
リクエスト受信
    ↓
信頼度を評価
    ├── HIGH（方向性が明確）
    │     → 5要素を揃えてCoSに指示
    │     → CoSが実行層にアサイン
    │
    ├── MEDIUM（概ね分かるが詳細が不明）
    │     → ドラフト/プロトタイプを作成
    │     → ユーザーと壁打ちで方向性確定
    │     → HIGH に昇格してから実行
    │
    └── LOW（何をすべきか不明確）
          → 選択肢A/B/Cを整理して提示
          → ユーザーと共同で方向決定
          → その後実行フローへ
```

**最重要ルール**: LOWやMEDIUMの状態で実行層に流さない。確信がないまま実行すると手戻りが発生する。

## 5要素チェック（CoSのゲート）

CoSはCEOから指示を受けたとき、以下の5要素が揃っているか確認する。1つでも欠けていたらCEOに差し戻す：

| # | 要素 | 例 |
|---|------|---|
| 1 | 誰のためか（具体的な人/チーム） | 「A社の田中部長向け」 |
| 2 | 何のためか（具体的な用途） | 「来週の提案プレゼン用」 |
| 3 | いつまでか（期限） | 「金曜17時まで」 |
| 4 | 方向性は確認済みか | CEO壁打ち済み ✓ |
| 5 | 品質基準は何か | 「そのまま使えるレベル」 |

**なぜこれが必要か**: 曖昧な指示を実行層に流すと、テンプレ的な成果物が出てきて使えない。具体的な文脈があって初めて実用的な出力になる。

## タスクルーティングフロー

```
ユーザーのDiscordメッセージ
    ↓
OpenClaw Gateway（Job ID発行）
    ├── キーワードマッチ → 特定エージェントへ
    └── デフォルト → CoSへ

CoSが受信：
    ├── ユーザーから直接？ → CEOにエスカレーション（戦略判断）
    └── CEOから？ → 5要素チェック
        ├── 完備 → 実行層にアサイン（Job ID付き）
        └── 不備 → CEOに差し戻し

実行層が完了：
    → ローカル保存
    → Notion Knowledge DBに記録
    → CEOに報告 → Discord #feed に通知（Notion URL付き）
```
