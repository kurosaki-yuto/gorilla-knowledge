# 正本DBの設計思想と失敗から学んだこと

## 原則: Single Source of Truth（SSOT）

**全てのデータの正本は1箇所（Supabase）に集約する。** Notionやスプレッドシートは「表示用」であり、正本ではない。

## データの流れ

```
エージェント → Supabase（正本DB）に書き込み
                    ↓ 同期スクリプト
              Notion（管理UI）に表示
```

**逆方向（Notion → Supabase）は禁止。** 人間がNotionで編集しても、それは正本に反映されない。

## スキーマ設計

5つのコアテーブル:

| テーブル | 用途 |
|---------|------|
| projects | プロジェクト管理 |
| agents | エージェント定義・状態 |
| jobs | タスク実行記録（Job ID付き） |
| outputs | 成果物記録 |
| alerts | アラート・通知記録 |

2つのビュー:
- `recent_jobs_summary`: 直近のジョブ一覧
- `agent_status_view`: エージェントの稼働状態

## スキーマ変更のルール

```
❌ Supabase SQL Editor で直接 ALTER TABLE
✅ schema.sql を更新 → レビュー → SQL Editor で実行
```

スキーマ定義ファイル（`schema.sql`）を経由することで：
- 変更履歴がgitに残る
- 別環境への再現が容易
- レビューが可能

## 失敗事例

### 1. NotionをDBとして使ってしまった

初期は「Notionに全部書けば楽」と思ってNotion をデータの正本にしていた。結果：
- API制限（3リクエスト/秒）でバッチ処理が詰まる
- プロパティの型制約が弱く、不正データが入る
- 検索が遅い
- インテグレーションの共有設定を忘れて全DB 404（別の失敗事例参照）

**教訓**: Notionは人間向けUIとして優秀だが、プログラムからの読み書きには向かない。

### 2. 複数箇所にデータが分散した

Supabase、Notion、Discord、ローカルファイルにデータが散らばり、「どこが正しいのか分からない」状態になった。

**教訓**: 正本は1箇所に決める。他は全て同期表示。書き込み先は正本DBのみ。

### 3. スキーマを直接変更して巻き戻せなくなった

開発中にSQL EditorでALTER TABLEを直接実行。後から「何を変えたか」が分からなくなった。

**教訓**: 面倒でも `schema.sql` を経由する。gitで差分が追える。
