# 5層アーキテクチャ（入口→統括→実行→DB→UI）

## 概要

AIエージェントシステムを5つの層に分離する設計パターン。各層の責務を明確に分けることで、複雑さを管理する。

## 5層の定義

```
┌─────────────────────────────────────┐
│  Layer 1: 入口（Gateway）            │
│  指示受付・ルーティング・通知返却      │
│  ※状態を持たない・重い処理をしない     │
├─────────────────────────────────────┤
│  Layer 2: 統括（Orchestrator）       │
│  意思決定・タスク分解・アサイン        │
│  ※実装はしない                       │
├─────────────────────────────────────┤
│  Layer 3: 実行（Executor）           │
│  コード生成・ファイル操作・API呼出     │
│  ※判断はしない                       │
├─────────────────────────────────────┤
│  Layer 4: 正本DB（Database）         │
│  全データの唯一の真実の源泉           │
│  ※アプリケーションロジックを持たない   │
├─────────────────────────────────────┤
│  Layer 5: 管理UI（Dashboard）        │
│  人間向け表示・ダッシュボード          │
│  ※正本ではない。DBからの同期表示      │
└─────────────────────────────────────┘
```

## 各層のルール

### Layer 1: 入口
- 5秒以上かかる処理をさせない
- 長文コンテキストを保持しない
- やること：指示受付、Job ID発行、ルーティング、通知返却

### Layer 2: 統括
- コードを書かない、ファイルを直接操作しない
- やること：優先順位判断、タスク分解、アサイン、進捗管理

### Layer 3: 実行
- 意思決定をしない（不明点は統括層に返す）
- やること：コード生成、データ操作、API連携、ファイル出力

### Layer 4: 正本DB
- アプリケーションロジックを入れない
- スキーマ変更は定義ファイル経由で行う（直接ALTER禁止）
- やること：データの永続化、整合性保証

### Layer 5: 管理UI
- ここにデータを書き込まない（表示専用）
- 正本DBからの同期スクリプトで更新する
- やること：ダッシュボード表示、人間向けレポート

## なぜこの分離が必要か

| 分離しなかった場合の問題 | 例 |
|------------------------|---|
| Gatewayに重い処理 | 入口が詰まって全リクエストが止まる |
| 統括層がコードを書く | 高コストなモデルで大量のコード生成 → 費用爆発 |
| 実行層が判断する | 方針のブレ、人間の意図と乖離した実装 |
| UIをDBとして使う | データ不整合、同期の地獄 |

## 実運用からの教訓

- **「UIは正本ではない」を徹底する**。Notionに直接データを書き込み始めると、DBとNotionでデータがズレて地獄になる
- **Gatewayは薄く保つ**。状態管理をGatewayに持たせると、再起動で全部消える
- **Job IDで全てを追跡する**（形式: `JOB-YYYYMMDD-XXXX`）。IDがないタスクは追跡不能になり、失敗時の原因調査ができない
